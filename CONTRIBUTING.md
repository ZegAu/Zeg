# Contributing to Zeg

Thanks for your interest in contributing. This document explains how we handle issues, pull requests, and code contributions.

Reporting issues
- Use the issue templates (bug or feature) to provide consistent information, including steps to reproduce, environment, and versions.

Pull requests
- Fork and create a branch named `fix/short-description` or `feat/short-description`.
- Include tests for new features or bug fixes when applicable.
- Keep PRs small and focused.
- Use clear commit messages (imperative tense): "Add X", "Fix Y".
- Link related issues in the PR description.

PR checklist
- [ ] Tests added/updated
- [ ] Linters/formatters run
- [ ] Documentation updated
- [ ] Changelog entry (if applicable)

Response times
- We aim to respond to issues and PRs within 72 hours during business days.
- Critical/security items follow .github/SECURITY.md.

Development environment
- Use the project-standard toolchain (we document runtime/tool versions in the repo).
- Run tests and linters before submitting PRs.
